A stacked diff should answer three questions at once. Which real Git commit am I reviewing? What changed in the GraphReFly graph? And which exact lines produced that change?

This is GraphReFly Stack reviewing a small, ordinary repository—not a prebuilt flagship fixture. The public command takes a repository, base, and head. It discovers three linear commits from real Git history. Version one is intentionally narrow: one root package, an unchanged lockfile, and an exact GraphReFly zero point three point zero pin.

For every revision, the repository itself emits a GraphBlueprint version two. Stack does not hand-write this picture. It uses the target GraphReFly runtime's public parse, hash, Blueprint-to-Mermaid, and diff helpers. Here, one commit adds a derived node and its edge.

Click the next commit. The commit O I D, Blueprint hash, diagram, parent delta, and code comparison all move together. Mounting the audit graph produces a subgraph-added event and a node-added event. Below it is the exact parent diff in a familiar split view—not raw terminal output.

The final commit changes only metadata. The graph delta says node-changed: audit status. The code panel shows the one changed line that adds the validated observer role. That correspondence is generated from the checked-out revision, not copied from a golden artifact.

The first usable release is deliberately honest about its boundary. Revisions run in detached, read-only, network-denied worktrees with an empty environment, a five-second timeout, a one-mebibyte output cap, and a sixty-four-commit limit. Missing configuration, dependency drift, malformed Blueprints, diagnostics errors, and hash mismatch all fail closed. User-relevant provenance is collapsed. With no semantic policy configured, the product says exactly that; it does not invent a green gate.
